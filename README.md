##Задание:                                                                                                                                                                                                                                                                                **Есть несколько (N) nginx в режиме обратного прокси (proxy_pass...). За ними приложение, отвечающее по HTTP.**
**Важно! Запрос от пользователя может прийти на любой из nginx. Запрос от пользователя может пройти как через один nginх, так и по цепочке через несколько nginx.**
**Примеры:**
```
пользователь>nginx1>приложение
пользователь>nginx2>приложение
...
пользователь>nginxN>приложение
пользователь>nginx1>nginx2>nginxN>приложение
пользователь>nginx2>nginxN>приложение
```
Приложение проверяет заголовок X-Forwarded-For.
Требуется, чтобы приложение в заголовке получило IP адрес пользователя и всех nginx (всю цепочку), через которые прошел запрос.
Требуется, чтобы приложение не получило заголовок X-Forwarded-For, который недобросовестный пользователь может добавить в свой запрос.

Решение подготовить в виде тестового стенда для запуска в docker-compose минимум из трех серверов nginx и любого готового или своего приложения выводящего заголовок X-Forwarded-For.
Предоставить протокол тестирования на выполнение всех требований задания с помощью утилиты  curl.**

*******************************************************************************************************************************************
Как это работает на одном сервере:
++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++
Я создала директорию test, в ней в трех директориях: nginx1, nginx2, nginx3 конфиги nginx (nginx.conf)
****************************************************************************************************
nginx1 (балансировщик) принимает запросы от клиента и решает куда отправить запрос дальше, поэтому у него используется upstream из нескольких серверов:
```
upstream backend {
    server nginx2:80;
    server nginx3:80; 
}
```
Это означает, что 
nginx1 может проксировать:
- либо в nginx2
- либо в nginx3, то есть nginx1 выполняет роль reverse proxy, балансировщика нагрузки
**************************************************************************************************************************************************nginx3 - последний reverse proxy перед приложением, он не балансирует, он просто передает запрос в приложение:
nginx3 -> app, поэтому
```                                                                                                                                                                                   upstream app_backend {
    server app:80;
}
```
+++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++
Вариант балансировки:
client
  ↓
nginx1
 ↙     ↘
nginx2 nginx3
  ↓       ↓
 app     app
Тогда nginx1 имеет upstream из нескольких nginx

Вариант цепочки proxy
client
  ↓
nginx1
  ↓
nginx2
  ↓
nginx3
  ↓
 app
Тогда nginx1 отправляет запрос дальше в nginx2
server nginx2:80;

nginx2 отправляет дальше запрос в nginx3
server nginx3:80;

nginx3 отправляет запрос уже в приложение
server app:80;

Иными словами коротко: Первый nginx выполняет роль entrypoint reverse proxy и балансировщика нагрузки, поэтому у него upstream содержит несколько nginx-серверов.
Последний nginx в цепочке является reverse proxy непосредственно перед приложением, поэтому его upstream указывает уже на app:80.

Как проверить, что приложение проверяет заголовок, приложение в заголовке получает IP адрес пользователя и всех nginx (всю цепочку), через которые прошел запрос.
```

irina@kubbernetes:~/test/nginx3$ curl http://176.123.160.109:8081
Hostname: b9c290c2c89e
IP: 127.0.0.1
IP: ::1
IP: 172.19.0.2
RemoteAddr: 172.19.0.3:41158
GET / HTTP/1.1
Host: 176.123.160.109
User-Agent: curl/8.5.0
Accept: */*
X-Forwarded-For: 172.19.0.1, 172.19.0.5
```

запрос пошёл по более короткому маршруту: client -> nginx1 -> nginx3 -> app, поэтому приложение увидело: X-Forwarded-For: 172.19.0.1, 172.19.0.5
в nginx1 настроен upstream с балансировкой:
```
upstream backend {
    server nginx2:80;
    server nginx3:80;
}
```
nginx1 сам выбирает:

либо nginx2
либо nginx3

Поэтому получаются разные цепочки
Иногда:
client -> nginx1 -> nginx2 -> nginx3 -> app

```
irina@kubbernetes:~/test/nginx3$ curl http://176.123.160.109:8081
Hostname: b9c290c2c89e
IP: 127.0.0.1
IP: ::1
IP: 172.19.0.2
RemoteAddr: 172.19.0.3:40278
GET / HTTP/1.1
Host: 176.123.160.109
User-Agent: curl/8.5.0
Accept: */*
X-Forwarded-For: 172.19.0.1, 172.19.0.5, 172.19.0.4
```

Запрос от пользователя может пройти:
- как через один nginx
- так и через цепочку из нескольких nginx
  
Поддельный X-Forwarded-For от клиента не попадает в приложение.
Проверяем:

```
irina@kubbernetes:~/test$ curl -s \
>   -H "X-Forwarded-For: 1.2.3.4" \
>   http://176.123.160.109:8081 | grep X-Forwarded-For
X-Forwarded-For: 172.19.0.1, 172.19.0.5
```

Каждый reverse proxy добавляет только доверенные адреса в цепочку X-Forwarded-For.
Можно увидеть все это на сервере с внешним IP 176.123.160.109 пароль 1978Happy$
